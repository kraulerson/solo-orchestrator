System under test: MCP server process, invoked via an MCP client. The scenarios below assume the MCP Inspector tool (npx @modelcontextprotocol/inspector) for reproducible JSON-RPC interaction. Claude Desktop or Claude Code are also valid clients — scenario output shape is the same, only the invocation wrapper differs.
Server command: python -m your_mcp_server (or the exact command the Orchestrator specified — it's in the project's mcp.json or equivalent). Run it in one terminal; Inspector in another.
Transport: stdio (default) OR HTTP (if the server is configured for it — scenarios that are HTTP-specific will say so).
Auth / environment: Export any required env vars before starting the server. Typically MCP_API_KEY=test-key and/or MCP_DB_URL=sqlite:///tmp/mcp-uat.db. The Orchestrator will list the exact set in the dispatch email.
Required tools: node (for MCP Inspector), python (or the server's runtime), jq (for inspecting JSON output).

One-time setup before any scenario:
1. Export the required env vars in the server terminal: export MCP_API_KEY=test-key (plus whatever else is specified).
2. Start MCP Inspector against the server: npx @modelcontextprotocol/inspector python -m your_mcp_server.
3. The Inspector UI opens in your browser. Confirm the server's tools and resources list loads without errors.

Every scenario below assumes:
1. The server is running (as in the setup).
2. MCP Inspector is open and connected.
3. If a scenario mutates server-side state (database rows, files), it includes cleanup steps.

How to run a scenario: click "details" on the scenario card, invoke the specified tool call or resource read in MCP Inspector, compare the JSON-RPC response to the "Expected" block, then click Pass / Fail / Skip. Paste the full response JSON into the Notes field if the scenario fails — that's the most useful thing to capture.
