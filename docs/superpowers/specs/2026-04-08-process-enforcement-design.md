# Process Enforcement & Tool Usage Tracking — Design Spec

## Version 1.0

**Date:** 2026-04-08
**Status:** Draft
**Scope:** Mechanical enforcement for sequential process compliance and MCP tool usage tracking during AI-directed development.
**Delivery:** One branch, one PR.

---

## Problem Statement

Real-world usage on the meshscope project exposed systematic process violations by the AI agent:

1. **UAT step skipping** — After finding 7 bugs in session 4, the agent jumped from test results directly to fixing and creating a PR, skipping 6 of 9 required steps (parallel agent dispatch, completeness verification, bug tracker consolidation, triage, TDD remediation, re-test, counter reset).

2. **Phase 2 initialization step skipping** — The agent skipped remote repo creation and branch protection (steps 1-2) and jumped to project scaffolding (step 3), inferring "local repo exists = steps complete."

3. **Context7 non-usage** — The agent never consulted Context7 for library documentation during an entire Phase 2 session involving PySide6 and VTK, relying on training data instead.

4. **Qdrant non-usage** — The agent never stored architecture decisions or debugging breakthroughs to Qdrant despite having it configured and being instructed to use it.

All four failures share the same root cause: **the instructions are clear but enforcement is Tier 3 (LLM compliance only)**. The agent knows the rules, follows them when it remembers, but skips them when in "flow" — and nothing mechanically prevents it.

**Audit of all sequential processes** identified 6 vulnerable process flows. Four are high/critical risk and warrant mechanical gating: Build Loop (Phase 2), UAT Session (Phase 2), Phase 3 Validation, and Phase 4 Release. Phase 2 Initialization is a one-time process gated by a verification check.

---

## Design Decisions (from brainstorming)

| Decision | Choice | Rationale |
|---|---|---|
| Enforcement approach | Process state machine + PreToolUse hook gate | Catches violations at commit time; agent must call checkpoint scripts to advance state |
| Tool usage tracking | PostToolUse hook + session-end reporting | Visibility without blocking mid-work; warns at commit time and session end |
| Commit type detection | Auto-detection (phase + staged files) | Simpler than agent declaration; agent can't misclassify |
| Context7/Qdrant enforcement level | Warn at commit, not block | Prevents false positives when agent legitimately doesn't need library docs |
| Process checklist failures | Block commit | Core enforcement mechanism — this is the point |
| Scope | Build Loop, UAT, Phase 3, Phase 4, Phase 2 Init | All high/critical risk processes; Phase gates already CI-enforced |

## 1. Process State Machine

### 1.1 State File

`.claude/process-state.json` tracks progress through all gated processes. Generated by init.sh with all processes at step 0.

```json
{
  "build_loop": {
    "feature": null,
    "step": 0,
    "steps_completed": [],
    "started_at": null
  },
  "uat_session": {
    "session_id": null,
    "step": 0,
    "steps_completed": [],
    "started_at": null
  },
  "phase3_validation": {
    "steps_completed": [],
    "started_at": null
  },
  "phase4_release": {
    "steps_completed": [],
    "started_at": null
  },
  "phase2_init": {
    "steps_completed": [],
    "verified": false
  }
}
```

### 1.2 Step Sequences

Each process has a defined, ordered step sequence. Steps must be completed in order — the checkpoint script validates that step N-1 is done before allowing step N.

**Build Loop (per feature, Phase 2):**

| # | Step ID | What It Means | Evidence |
|---|---|---|---|
| 1 | `tests_written` | Failing tests exist for this feature | Test files staged or committed |
| 2 | `tests_verified_failing` | Tests confirmed to fail (no implementation yet) | Agent attests test run showed failures |
| 3 | `implemented` | Code written, all tests pass | Test suite passes |
| 4 | `security_audit` | SAST + threat model review completed | Semgrep run, no unresolved findings |
| 5 | `documentation_updated` | CHANGELOG, FEATURES.md, Bible, ADRs, interface docs updated | Documentation files modified |
| 6 | `feature_recorded` | `test-gate.sh --record-feature` called | build-progress.json updated |

**UAT Session (Phase 2, Steps 2.7-2.9):**

| # | Step ID | What It Means |
|---|---|---|
| 1 | `agents_dispatched` | Parallel test agents (automated, exploratory, cross-platform) launched |
| 2 | `template_generated` | HTML test session file created in tests/uat/sessions/ |
| 3 | `orchestrator_notified` | Agent told orchestrator "your test template is at [path]" and is waiting |
| 4 | `results_received` | Orchestrator confirmed "results are in" |
| 5 | `completeness_verified` | All scenarios checked — incomplete scenarios listed |
| 6 | `bugs_consolidated` | All results written to BUGS.md |
| 7 | `triage_complete` | Orchestrator assigned dispositions to all bugs |
| 8 | `remediation_complete` | All Fix Now bugs resolved with test-first discipline |
| 9 | `gate_passed` | `test-gate.sh --check-batch` returns 0, counter reset |

**Phase 3 Validation:**

| # | Step ID | What It Means |
|---|---|---|
| 1 | `integration_testing` | E2E test suite run and passing |
| 2 | `security_hardening` | SAST, dependency scan, secret scan, license check, SBOM generated, threat model validated |
| 3 | `chaos_testing` | Edge case and error recovery tests run |
| 4 | `accessibility_audit` | UX and accessibility checks completed |
| 5 | `performance_audit` | Startup, latency, memory, bundle checks completed |
| 6 | `contract_testing` | Interface contract verification (Standard+ Track, skip for Light) |
| 7 | `results_archived` | All results saved to docs/test-results/ with naming convention |

**Phase 4 Release:**

| # | Step ID | What It Means |
|---|---|---|
| 1 | `production_build` | Builds successfully on all target platforms |
| 2 | `rollback_tested` | Rollback procedure tested and documented with time elapsed |
| 3 | `go_live_verified` | Go-live verification checklist completed on each platform |
| 4 | `monitoring_configured` | Error tracking and alerting verified |
| 5 | `handoff_written` | HANDOFF.md generated from template |

**Phase 2 Initialization:**

| # | Step ID | What It Means | Auto-Verifiable? |
|---|---|---|---|
| 1 | `remote_repo_created` | Remote origin exists and is accessible | Yes — check `.git/config` for remote |
| 2 | `branch_protection_configured` | Branch protection rules on main | Partial — check if CI workflow exists |

| 6 | `ci_pipeline_configured` | CI passes on first push | Partial — check workflow file exists |
| 7 | `initialization_verified` | All verification items pass | Composite of above |

## 2. Checkpoint Script

### 2.1 `scripts/process-checklist.sh`

Single script managing all process state.

**Commands:**

| Command | Purpose |
|---|---|
| `--start-feature "name"` | Reset build_loop to step 0, set feature name, set started_at |
| `--complete-step PROCESS:STEP_ID` | Mark step complete. Validates ordering — blocks if prior step not done. |
| `--start-uat N` | Initialize uat_session for session N at step 0 |
| `--start-phase3` | Initialize phase3_validation at step 0 |
| `--start-phase4` | Initialize phase4_release at step 0 |
| `--verify-init` | Run Phase 2 initialization verification. Auto-checks what it can, prompts for manual attestation on the rest. |
| `--status` | Print current state of all processes (human-readable) |
| `--check-commit-ready` | Return 0 if commit allowed, 1 if blocked. Output explains why. Used by the PreToolUse hook. |
| `--reset PROCESS` | Clear state for one process (emergency escape). Logs the reset to stderr. |
| `--reset-all` | Clear all process state. Emergency escape only. |

**Ordering enforcement in `--complete-step`:**

```
Input: --complete-step build_loop:security_audit

1. Look up build_loop step sequence: [tests_written, tests_verified_failing, implemented, security_audit, ...]
2. Find security_audit at index 3
3. Check steps_completed contains indices 0, 1, 2 (tests_written, tests_verified_failing, implemented)
4. If missing: exit 1 with "Cannot complete 'security_audit' — 'tests_verified_failing' not yet completed."
5. If present: add security_audit to steps_completed, update step to 4, exit 0
```

**`--check-commit-ready` logic:**

```
1. Read current_phase from .claude/phase-state.json
2. If phase < 2: exit 0 (no Build Loop enforcement)
3. If phase == 2 and phase2_init.verified == false:
     exit 1: "Phase 2 initialization not verified. Run: scripts/process-checklist.sh --verify-init"
4. Read staged files from git diff --cached --name-only
5. Classify commit:
   - Source extensions (.py, .ts, .js, .rs, .go, .cs, .kt, .java, .dart, .swift) or
     source directories (src/, lib/, app/) → source commit
   - Only .md, .json, .yml, .toml, .tmpl → docs commit → exit 0
   - No staged files → exit 0
6. For source commits in Phase 2:
   - If build_loop.feature is null: exit 1: "No feature started. Run: scripts/process-checklist.sh --start-feature 'name'"
   - If build_loop.steps_completed missing required steps up to documentation_updated:
     exit 1: "Build Loop step 'STEP' not completed. Run: scripts/process-checklist.sh --complete-step build_loop:STEP"
   - If uat_session is in progress (started_at != null) and not all 9 steps complete:
     exit 1: "UAT session N in progress — complete all steps before committing fixes."
7. For Phase 3: check phase3_validation steps
8. For Phase 4: check phase4_release steps (rollback_tested required before go_live_verified)
```

**`--verify-init` logic:**

For each Phase 2 initialization step, attempt auto-verification:
1. `remote_repo_created`: `git remote get-url origin` succeeds → auto-pass
2. `branch_protection_configured`: `.github/workflows/ci.yml` exists → auto-pass (best effort)
3. `project_scaffolded`: lockfile exists (check common names: `package-lock.json`, `Pipfile.lock`, `poetry.lock`, `Cargo.lock`, `go.sum`, `pubspec.lock`, `Package.resolved`) → auto-pass
4. `data_model_applied`: Cannot auto-verify → prompt: "Has the initial data model been applied and backup/restore tested? [y/n]"
5. `pre_commit_hooks_installed`: `.git/hooks/pre-commit` exists and is executable → auto-pass
6. `ci_pipeline_configured`: `.github/workflows/ci.yml` exists → auto-pass
7. `initialization_verified`: All above pass → set `verified: true`

Steps that fail auto-verification are listed. The agent must either fix them or attest manually for steps that can't be auto-checked.

## 3. PreToolUse Hook (Commit Gate)

### 3.1 `scripts/pre-commit-gate.sh`

Registered as a Claude Code `PreToolUse` hook on Bash tool calls.

**Activation:** The hook reads the bash command from stdin (Claude Code passes the tool input as JSON). It only activates when the command matches:
- `git commit` (any flags)
- `gh pr create` (any flags)

For all other commands, it exits immediately with no output (allow).

**For `git commit`:**

1. Run `scripts/process-checklist.sh --check-commit-ready`
2. If exit 1: output JSON to block: `{"hookSpecificOutput": {"hookEventName": "PreToolUse", "permissionDecision": "deny", "permissionDecisionReason": "REASON_FROM_SCRIPT"}}`
3. If exit 0: check tool usage (see Section 4)
4. If tool usage warnings exist: output as `additionalContext` (informational, not blocking)
5. If no issues: exit with no output (allow)

**For `gh pr create`:**

1. Same as git commit checks
2. Additionally: verify no UAT session is in progress with incomplete steps
3. Additionally: verify process-state build_loop is at step 0 or fully complete (no partial features)

**Auto-detection of commit type:**

The hook reads the current phase from `.claude/phase-state.json` and inspects `git diff --cached --name-only` to classify the commit. Documentation-only commits (all staged files are .md/.json/.yml/.toml/.tmpl) skip the process gate entirely. Source commits go through the full check.

Source file detection uses common patterns:
- Extensions: `.py`, `.ts`, `.tsx`, `.js`, `.jsx`, `.rs`, `.go`, `.cs`, `.kt`, `.java`, `.dart`, `.swift`, `.c`, `.cpp`, `.h`
- Directories: `src/`, `lib/`, `app/`, `pkg/`, `internal/`, `cmd/`

If any staged file matches, it's a source commit.

### 3.2 Hook Registration

In init.sh, register the PreToolUse hook in `.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash \"$CLAUDE_PROJECT_DIR\"/scripts/pre-commit-gate.sh"
          }
        ]
      }
    ]
  }
}
```

The hook receives the tool input (including the bash command) via stdin as JSON.

## 4. Tool Usage Tracking

### 4.1 State File

`.claude/tool-usage.json` tracks MCP tool calls per session.

```json
{
  "session_id": "2026-04-10T14:00:00Z",
  "calls": [
    {"tool": "mcp__context7__resolve-library-id", "timestamp": "2026-04-10T14:05:00Z"},
    {"tool": "mcp__context7__query-docs", "timestamp": "2026-04-10T14:05:30Z"},
    {"tool": "mcp__qdrant__qdrant-find", "timestamp": "2026-04-10T14:01:00Z"},
    {"tool": "mcp__qdrant__qdrant-store", "timestamp": "2026-04-10T14:45:00Z"}
  ],
  "commits_since_last_context7": 0,
  "qdrant_find_called": true,
  "qdrant_store_called": true
}
```

### 4.2 PostToolUse Hook — `scripts/track-tool-usage.sh`

Registered as a `PostToolUse` hook. Fires after every tool call.

**Logic:**
1. Read tool name from stdin (Claude Code passes tool result as JSON)
2. If tool name contains `context7`: append to calls, reset `commits_since_last_context7` to 0
3. If tool name contains `qdrant` and contains `find`: set `qdrant_find_called` to true, append to calls
4. If tool name contains `qdrant` and contains `store`: set `qdrant_store_called` to true, append to calls
5. If tool name is Bash and command contains `git commit`: increment `commits_since_last_context7`
6. Otherwise: exit immediately (no file I/O for non-MCP, non-commit tools)

**Performance:** The hook must be fast. For the vast majority of tool calls (Read, Write, Edit, Grep, Glob), it checks the tool name string and exits with no disk I/O. Only MCP tool calls and git commits trigger a file write.

### 4.3 Integration with Pre-Commit Gate

When `scripts/pre-commit-gate.sh` runs for a Phase 2 source commit, after the process checklist passes, it also checks `.claude/tool-usage.json`:

- **Context7:** If `commits_since_last_context7 >= 2` and staged files contain source code with library imports, output warning: `"Context7 has not been consulted for library documentation in the last 2 commits. Consider checking docs for libraries used in this change."`
- **Qdrant-find:** If `qdrant_find_called` is false (first commit of session only), output warning: `"No prior context retrieved from Qdrant this session. Consider checking for relevant architecture decisions and patterns."`

These are warnings (informational output), not blocks. The commit proceeds.

### 4.4 SessionStart Integration

Modify `scripts/session-test-gate-check.sh` (or add to existing SessionStart hooks):
- Create/reset `.claude/tool-usage.json` with a new `session_id` (current ISO timestamp)
- Empty `calls` array, zero counters, false flags

### 4.5 Stop Hook Integration

Modify `scripts/session-end-qdrant-reminder.sh`:
- Read `.claude/tool-usage.json`
- Count Context7 calls, Qdrant-find calls, Qdrant-store calls
- Output summary: `"This session: Context7 called N times, Qdrant-find N times, Qdrant-store N times."`
- If current phase is 2 and source commits were made this session:
  - If `qdrant_store_called` is false: warn — `"You made source commits this session but stored nothing in Qdrant. Before ending, store any architecture decisions, debugging breakthroughs, or integration patterns."`
  - If Context7 was never called: warn — `"Source code was committed but Context7 was never consulted. If you used library APIs, check Context7 for current documentation next session."`

### 4.6 Hook Registration

In init.sh, register the PostToolUse hook in `.claude/settings.json`:

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "bash \"$CLAUDE_PROJECT_DIR\"/scripts/track-tool-usage.sh"
          }
        ]
      }
    ]
  }
}
```

## 5. Phase 2 Initialization Gate

### 5.1 One-Time Verification

The `phase2_init` section of `.claude/process-state.json` is checked when `--start-feature` is called for the first time. If `verified` is false, the script blocks and directs the agent to run `--verify-init`.

### 5.2 Auto-Verification Checks

| Step | Check | Method |
|---|---|---|
| `remote_repo_created` | Remote origin exists | `git remote get-url origin` succeeds |
| `branch_protection_configured` | CI workflow exists | `.github/workflows/ci.yml` file check |
| `project_scaffolded` | Lockfile exists | Check for common lockfile names |
| `data_model_applied` | Cannot auto-verify | Prompt agent for manual attestation |
| `pre_commit_hooks_installed` | Pre-commit hook exists | `.git/hooks/pre-commit` exists and is executable |
| `ci_pipeline_configured` | CI workflow exists | `.github/workflows/ci.yml` file check |
| `initialization_verified` | All above pass | Composite |

Auto-verified steps are marked complete silently. Steps requiring attestation prompt: `"Cannot auto-verify: [step description]. Has this been completed? [y/n]"` Agent must respond.

## 9. Emergency Escape

If the enforcement system blocks a legitimate action due to a bug or edge case:

- `scripts/process-checklist.sh --reset build_loop` — clears one process
- `scripts/process-checklist.sh --reset-all` — clears everything

Resets are logged to stderr with a timestamp so they're visible in the session history. The Orchestrator can see if the agent used an escape hatch.

The `--reset` commands should not be in the CLAUDE.md template instructions. They exist for the Orchestrator to use manually if needed, not for the agent to self-invoke.
