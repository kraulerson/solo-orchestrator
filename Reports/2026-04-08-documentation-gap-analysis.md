## Executive Summary

The Solo Orchestrator Framework is meticulous about specifying *what* documentation artifacts must exist and *when* they are produced. It is significantly less meticulous about specifying *how* they should be structured internally. The framework demands 16+ documentation artifacts per project (Appendix A of the Builder's Guide lists them), provides templates for exactly 4 of them (CLAUDE.md, APPROVAL_LOG.md for both personal and organizational, UAT test template, and PROJECT_INTAKE.md), and leaves the remaining 12+ artifacts to be generated by the AI agent with no structural template, no naming convention for most, and no specification of where within `docs/` they should live.

This creates a paradox: a framework that talks extensively about documentation discipline -- "Every phase produces documentation enabling a qualified replacement to resume maintenance" -- does not provide the structural scaffolding to make that documentation consistent across projects. Two different Orchestrators building two different projects with this framework will produce PROJECT_BIBLE.md files with different internal structures, ADRs stored in different locations (or not at all), interface documentation in different formats, and CHANGELOG.md files with different conventions. The documentation *will exist* because the framework demands it. It will not be *consistent* because the framework does not template it.

**Total gaps identified: 15**
- Critical (blocks consistent execution): 5
- Moderate (causes inconsistency across projects): 7
- Minor (cosmetic/polish): 3

## Gap Analysis

### 1. Architecture Decision Records (ADRs)

**Current State:**

ADRs are mentioned in five places with varying levels of specificity:

1. **Builder's Guide, line 641 (Step 1.6, item 3):** "Architecture Decision Record (selected stack, rejected alternatives, rationale)" -- listed as one of 16 items in the Project Bible. This is the Phase 1 ADR, embedded inside PROJECT_BIBLE.md.
2. **Builder's Guide, line 889 (Step 2.5):** "Architecture/UX Decision Record: For non-trivial decisions." -- listed as a per-feature Phase 2 documentation output. No specification of where this goes.
3. **Builder's Guide, line 1432 (Appendix A):** "Architecture Decision Records | 1-2 | Every major choice with alternatives and rationale" -- listed as a project artifact spanning Phases 1-2.
4. **User Guide, line 700:** "What you produce: Approved PROJECT_BIBLE.md, CONTRIBUTING.md, Architecture Decision Records" -- listed as a Phase 1 output.
5. **CLI Setup Addendum, line 314 (Qdrant section):** "Phase 2 Architecture Decision Records: Architecture decisions are stored semantically." -- mentions ADRs as something stored in Qdrant for retrieval.
6. **CLAUDE.md template, line 88:** "Phase 2.5 Documentation: Parallel generation of CHANGELOG, interface docs, and ADRs" -- instructs the agent to generate ADRs as a parallel task.

**What is NOT specified:**
- Where do standalone ADRs live? There is no `docs/adr/` or `docs/decisions/` directory created by `init.sh` (line 1023 creates only `docs/framework`, `docs/platform-modules`, `docs/test-results`).
- What format should ADRs use? No mention of the standard ADR template (Title, Status, Context, Decision, Consequences) or any alternative.
- How are ADRs named or numbered? No convention specified (e.g., `0001-use-postgresql.md` vs. `adr-postgresql.md` vs. embedded in the Bible).
- How does the initial Phase 1 ADR (embedded in the Project Bible under item 3) relate to ongoing Phase 2 ADRs? Are Phase 2 ADRs also embedded in the Bible? Are they separate files? The Bible says "Architecture Decision Record" (singular, embedded), while Appendix A says "Architecture Decision Records" (plural, listed as a separate artifact).
- There is no ADR template in `templates/`.

The Phase 1 ADR is embedded in PROJECT_BIBLE.md. The Phase 2 ADRs float in specification limbo -- Step 2.5 says to produce them, but does not say where to put them or what they look like.

**Gap Severity: Critical**

**Impact:** Without a defined location, format, or naming convention, Phase 2 ADRs will either be embedded haphazardly into the Bible (making it grow without structure), dropped into the project root, or lost entirely. Different Orchestrators will handle this differently. The handoff document (HANDOFF.md) has no way to reference ADRs consistently because there is no consistent location. Qdrant storage is not a substitute for file-based ADRs -- it is supplementary semantic retrieval, not the canonical record.

**Remediation Options:**

1. **Standard ADR directory + template (recommended).** Create `docs/decisions/` in `init.sh`. Provide `templates/generated/adr.tmpl` with the standard ADR format (Title, Status, Context, Decision, Consequences). Number sequentially: `0001-title.md`. The Phase 1 ADR remains embedded in the Bible but is also the first ADR in `docs/decisions/0001-architecture-selection.md`. Phase 2 ADRs are created per-feature when non-trivial decisions occur. **Tradeoff:** Adds a directory and template; adds ~30 lines to `init.sh`. Minimal overhead, maximum consistency.
2. **Embedded ADR section in the Bible.** Define a structured section within PROJECT_BIBLE.md where all ADRs are appended in a consistent format. **Tradeoff:** Keeps everything in one file but the Bible will grow unbounded. For small projects this is fine; for large projects the Bible becomes unwieldy.
3. **ADRs as CHANGELOG entries.** Treat ADRs as a specialized CHANGELOG entry type. **Tradeoff:** Loses the decision record structure (Context, Alternatives, Consequences) and conflates two different document types.

### 2. Interface Documentation

**Current State:**

- **Builder's Guide, line 888 (Step 2.5):** "Interface Documentation: Every new API endpoint, command, or user-facing interface with contracts and error codes."
- **Builder's Guide, line 1435 (Appendix A):** "Interface Documentation | 2+ | Per-endpoint/command/UI contracts, error codes"
- **User Guide, line 794:** "interface documentation (API endpoints, contracts, error codes)"

**What is NOT specified:**
- Where does interface documentation live? No directory or file specified. Is it a single `API_DOCS.md`? Is it in `docs/api/`? Is it auto-generated (e.g., OpenAPI/Swagger)?
- What format should it use? The framework mentions "contracts and error codes" but does not specify whether this is OpenAPI spec, a Markdown table, JSDoc/pydoc inline, or a separate specification file.
- There is no template for interface documentation.
- The User Guide's Phase 2 Build Loop (line 794) mentions it in passing: "interface documentation (API endpoints, contracts, error codes)" -- no further detail.

**Gap Severity: Moderate**

**Impact:** The format and location of interface documentation varies wildly by project type. A web API might generate OpenAPI specs. A desktop app might document IPC commands. A CLI tool documents subcommands and flags. Without guidance, the AI agent will choose something reasonable but inconsistent across projects. The handoff document cannot reliably reference interface documentation because there is no canonical location.

**Remediation Options:**

1. **Platform-specific guidance in Platform Modules (recommended).** Add a "Documentation Conventions" section to each Platform Module specifying the interface documentation format and location for that platform. Web: OpenAPI spec at `docs/api/openapi.yml` or `docs/api/` directory. Desktop: `docs/interfaces.md` for IPC commands. Mobile: `docs/api/` for backend API contracts. CLI: `docs/commands.md` with man-page-style format. **Tradeoff:** Requires updating 3 Platform Modules. Most natural fit since interface documentation is platform-dependent.
2. **Generic `docs/interfaces/` directory.** Create the directory in `init.sh` and let the agent populate it. Provide a minimal template. **Tradeoff:** One-size-fits-all may not fit any platform well.
3. **Embed in the Bible.** Add interface contracts as a section in PROJECT_BIBLE.md that grows per-feature. **Tradeoff:** Same unbounded growth problem as embedding ADRs.

3. **Merge with ADRs.** If "Feature Documentation" is really "why we built this feature this way," it is an ADR variant. Fold the Appendix A description into the ADR template as optional sections. **Tradeoff:** May overload the ADR format.

### 6. Project Bible (PROJECT_BIBLE.md) Structure

**Current State:**

- **Builder's Guide, lines 638-682 (Step 1.6):** Lists 16 items that must be in the Project Bible:
  1. Full Product Manifesto text
  2. Revenue Model & Cost Constraints
  3. Architecture Decision Record
  4. Threat Model & Risk/Mitigation Matrix
  5. Data Model
  6. Data Migration Plan
  7. Auth & Identity Strategy
  8. Observability & Logging Strategy
  9. UI Component Specifications
  10. Coding Standards
  11. Build & Distribution Strategy
  12. Test Strategy
  13. Orchestrator Profile Summary
  14. Accessibility Requirements
  15. Platform-Specific Requirements
  16. Context Management Plan

  Plus a Bug Severity Classification table and a UAT Plan.

**What is NOT specified:**
- No template for PROJECT_BIBLE.md. This is a 16-section document with embedded sub-structures (Bug Severity table, UAT Plan) and no template.
- No guidance on section ordering beyond the numbered list.
- No indication of expected length or depth for each section.
- The Bible is supposed to be updated every feature (Step 2.5: "Project Bible Update: New interfaces, data model changes, new configuration, new dependencies"), but no guidance on how to structure updates versus the original content.
- The Context Management Plan (item 16) defines Bible management tiers (small/medium/large projects) but no mechanism for actually creating the condensed versions.

**Gap Severity: Critical**

**Impact:** The Project Bible is the most important artifact in the framework. It is loaded into context at every session start. It gates Phase 1-to-Phase 2 transition. It is referenced during every security audit, every feature build, and every context health check. A 16-section document without a template will be structured differently by every AI agent on every project. The agent will produce all 16 items because the prompt demands them, but the internal structure, heading levels, section depths, and organization will vary. This makes the Context Management Plan (item 16) even harder to implement -- you cannot condense a document whose structure you cannot predict.

**Remediation Options:**

1. **Provide a comprehensive template in `templates/generated/project-bible.tmpl` (recommended).** Include all 16 sections as Markdown headings with placeholder descriptions. Include the Bug Severity Classification table and UAT Plan subsections pre-formatted. Add `<!-- UPDATED: YYYY-MM-DD -->` markers per section for tracking freshness. **Tradeoff:** Large template (~200 lines). But this is the single most impactful template the framework could provide.
2. **Provide a structural outline in Step 1.6.** List sections with required heading levels and expected content type (table, prose, list). No separate file. **Tradeoff:** Makes the Builder's Guide longer but keeps everything in one place.
3. **Provide a Bible Index template.** Instead of templating the full Bible, template the condensed index (item 16, "Condensed Bible Index under 5,000 tokens") that is used for large projects. **Tradeoff:** Does not solve the structural inconsistency of the full Bible.

### 12. docs/ Directory Structure

**Current State:**

```
docs/framework/         -- framework reference documents (Builder's Guide, Governance, etc.)
docs/platform-modules/  -- platform-specific guidance
docs/test-results/      -- Phase 3 test result archives (empty at init)
```

The Builder's Guide specifies these additional documentation locations:
- `docs/INCIDENT_RESPONSE.md` (Step 4.1.5, line 1283)
- `docs/test-results/[date]_[type]_[pass|fail].[ext]` (Step 3.5.9)

Artifacts specified but with NO location guidance:
- Architecture Decision Records (Phases 1-2)
- Interface Documentation (Phase 2+)
- Feature Documentation (Phase 2+, Appendix A only)
- Security Audit Logs (Phase 3, Appendix A)
- Performance Baselines (Phase 3, Appendix A)

Root-level artifacts (well-defined):
- `PRODUCT_MANIFESTO.md`
- `PROJECT_BIBLE.md`
- `CONTRIBUTING.md`
- `CHANGELOG.md`
- `CLAUDE.md`
- `APPROVAL_LOG.md`
- `PROJECT_INTAKE.md`
- `HANDOFF.md`
- `RELEASE_NOTES.md`
- `USER_GUIDE.md`
- `sbom.json`

**Gap Severity: Critical**

**Impact:** The expected directory structure for project documentation is partially defined (root-level files and `docs/test-results/`), partially implicit (`docs/INCIDENT_RESPONSE.md` is mentioned in text), and partially undefined (ADRs, interface docs, feature docs, security audit logs, performance baselines). ### 15. UAT Test Session Format and Lifecycle

**Current State:**

The framework provides a Markdown template (`templates/uat-test-template.md`) for UAT test sessions. The Builder's Guide (Step 2.7, line 917) instructs the agent to generate a test template and place it in `tests/uat/sessions/<date>-session-N/templates/`.

Real-world usage on the meshscope project exposed three failures in this approach:

1. **Sessions 1-3:** UAT test plans were output to the terminal screen only -- no file artifact was created. The framework says to generate a file (Step 2.7, line 917) but sessions 1-3 produced no persistent record. The agent defaulted to screen output because nothing enforced file creation.
2. **Session 4:** The first time a file was created (`tests/uat/sessions/session-4-f7-f8/test-session-4.md`), but Markdown tables with empty `Pass/Fail` cells are hostile to human input. Users must manually type "Pass" or "Fail" into table cells, navigate whitespace-sensitive Markdown formatting, and cannot interact with the document naturally.
3. **HTML replacement:** The Orchestrator had the agent recreate the test session as `test-session-4.html` -- an interactive HTML file with Pass/Fail/Skip buttons, expandable test details, per-scenario notes fields, a bug entry form with severity dropdowns, a progress bar, and a "Copy Results to Clipboard" export function. This was immediately usable.

**What is NOT specified:**

- **Format:** The framework assumes Markdown but does not specify it. The Markdown template is the wrong format for human interaction.
- **HTML template:** No HTML UAT template exists in `templates/`. The meshscope HTML was generated ad hoc by the agent.
- **Versioning:** No convention for re-tests. When a UAT session is run again after bug fixes, should it overwrite v1? Create v2? The meshscope session has no version suffix.
- **Archival:** No guidance on moving completed UAT sessions to `docs/test-results/` for historical reference alongside the Phase 3 scan archives. UAT results live in `tests/uat/sessions/` permanently -- separate from the other test evidence.
- **Consistency:** Without an HTML template, each agent-generated HTML file will have different styling, different interactive controls, and different export formats. The meshscope HTML is excellent but it was a one-off creation, not a reusable template.

**Gap Severity: Critical**

**Impact:** UAT testing is the primary human validation mechanism in the framework. If the test artifact format is hostile to human input (Markdown tables), testers will skip scenarios, provide minimal notes, or avoid the process entirely. The first three sessions on meshscope produced no file artifact at all -- three UAT sessions with zero historical record. The framework's test-fix-verify loop (Steps 2.7-2.9) depends on UAT quality, and UAT quality depends on the test format being usable.

The versioning gap means re-tests after bug fixes overwrite the original results, destroying the evidence that bugs existed and were resolved. For organizational deployments with audit trail requirements, this is a compliance gap.

**Remediation Options:**

1. **HTML as the standard UAT format with a reusable template (recommended).**
   - Create `templates/uat-test-session.html` -- a self-contained HTML template with interactive controls (Pass/Fail/Skip buttons, notes fields, bug entry form, progress tracking, export-to-clipboard). The template includes JavaScript placeholder patterns that the agent populates with feature-specific scenarios at session generation time.
   - **Naming convention:** `test-session-N-v1.html`. Version increments on re-test after bug fixes (v2, v3, etc.). Original versions are never overwritten.
   - **Session directory:** `tests/uat/sessions/session-N-fX-fY/` (already specified in Step 2.7, matches meshscope usage).
   - **Archival:** After completion and agent review, the completed HTML is copied to `docs/test-results/` for permanent historical reference, named: `[date]_uat-session-N-vX.html`. This places UAT evidence alongside Phase 3 scan archives.
   - **Export:** The HTML must include a "Copy Results" function that exports structured Markdown for pasting into the agent conversation or saving to `submissions/`.
   - **Tradeoff:** Requires building and maintaining an HTML template (~300 lines). Significantly more complex than a Markdown template. But the meshscope experience proves the usability improvement is not optional.

2. **Keep Markdown but fix the template for input.** Redesign the Markdown template to use checkbox lists (`- [ ] Pass / - [ ] Fail`) instead of table cells. Easier to fill in but still not interactive -- no progress tracking, no export, no bug form. **Tradeoff:** Minimal effort but does not solve the core usability problem. Still requires testers to edit a text file.

3. **Dual format -- HTML for testing, Markdown for archival.** Generate HTML for the active testing session, export completed results as Markdown for archival. **Tradeoff:** Both formats maintained. The HTML is the working copy; the Markdown export is the permanent record. This may be the best hybrid if HTML files are awkward in `docs/test-results/` alongside JSON scan results.

**Document types:**
- **Governing documents** (Manifesto, Bible): Written once, updated in-place. The agent must verify consistency after every update.
- **Append-only logs** (CHANGELOG, APPROVAL_LOG, RELEASE_NOTES): New entries prepended or appended. Previous entries never modified.
- **Decision records** (ADRs): Write-once. If a decision is reversed, a new ADR is created with Status: Superseded referencing the original.
- **Snapshots** (test results, SBOM): Point-in-time artifacts. Named with date. Never modified after creation.
- **Reference documents** (HANDOFF, USER_GUIDE, INCIDENT_RESPONSE): Written at Phase 3-4, updated during maintenance. Should have "Last Updated" markers.

## Critical Assessment

### Is this a framework that talks about documentation discipline but does not practice it in its own artifact definitions?

Partially, yes.

The framework is *extremely* disciplined about specifying *that* documentation must exist, *when* it must be created, and *why* it matters. The Builder's Guide's Appendix A, the phase gate requirements, the CI changelog freshness check, the Context Health Check, the HANDOFF.md handoff test, and the approval log system are all evidence of a framework author who takes documentation seriously.

The gap is in the *structural* discipline. The framework demands a 16-section Project Bible but provides no template. It demands ADRs but specifies no format or location. It demands interface documentation but leaves format to the platform-dependent whim of the AI agent. It demands a CHANGELOG but does not specify which of the three major changelog formats to use.

This is not hypocrisy -- it is a specific class of omission. The framework author clearly understands documentation requirements at the process level (what, when, why) but has not extended that rigor to the structural level (how, where, in what format). The result is a framework that guarantees documentation *existence* but not documentation *consistency*.

For a solo-use framework, this is tolerable -- one Orchestrator will develop internal consistency across their own projects. For an organizational framework with multiple Orchestrators (which the Governance Framework explicitly contemplates with portfolio governance, quarterly reviews, and graduation criteria), this is a structural gap that will produce inconsistent documentation across the portfolio.

### Minimum Viable Fix

1. **Fix Appendix A** (Gap #14). Add missing artifacts, remove Feature Documentation, clarify ambiguous entries. This is a text edit to ~15 lines in the Builder's Guide. Estimated effort: 30 minutes.
2. **Add `project-bible.tmpl`** (Gap #6). This is the single highest-impact template. A consistent Bible structure makes every downstream process more reliable -- context loading, health checks, handoff, auditing. Estimated effort: 2-3 hours.
3. **Add `adr.tmpl` and `docs/decisions/` directory** (Gap #1). Standard ADR format (Title, Status, Context, Decision, Consequences). ~5 lines in `init.sh` to create the directory. ~30 lines for the template. Estimated effort: 1 hour.
4. **Specify CHANGELOG format** (Gap #7). Add one sentence to Step 2.5: "Use Keep a Changelog format." Estimated effort: 10 minutes.
5. **Replace `uat-test-template.md` with `uat-test-session.html`** (Gap #15). Build an interactive HTML template with Pass/Fail/Skip buttons, notes fields, bug entry form, progress bar, and export-to-clipboard. Define versioning convention (`test-session-N-v1.html`) and archival step (completed sessions copied to `docs/test-results/`). Update Step 2.7 and CLAUDE.md template to reference HTML format. Estimated effort: 3-4 hours.

**Total minimum viable fix: ~8-10 hours of work. No new concepts. No process changes. Just structural scaffolding for artifacts the framework already demands, plus fixing the UAT format that real-world usage proved is broken.**

### Ideal Fix

All items in the "Proposed Template Set" table above, the canonical docs/ directory structure added to `init.sh`, Appendix A reconciliation, per-section freshness markers in the Bible template, and the Phase 3 completeness check for `docs/test-results/`. The UAT HTML template should be parameterized so the agent can populate it with project-specific scenarios, feature names, and test fixture references.

**Total ideal fix: ~3-4 days of work. Still no new concepts or process changes -- purely structural scaffolding and cross-reference reconciliation.**
