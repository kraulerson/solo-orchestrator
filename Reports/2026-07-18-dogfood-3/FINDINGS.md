## F-DF3-001 (Medium) — Phase-0 approval "placeholder" detector self-collides with the framework's own template: filling Approval History per the template makes the FIRST gate unpassable
- **Where:** `scripts/check-phase-gate.sh::validate_approval_fields` — `section=$(grep -A 20 "$gate_name" "$APPROVAL_LOG")` then `grep -qiE "(Approver|Reviewer).*\[.*\]|YYYY-MM-DD"` (grep-able: `placeholder values`).
- **Mechanism (two stacked precision bugs):**
  1. `grep -A 20 "Phase 0.*Phase 1"` matches EVERY line containing the gate name — the `## Phase Gate:` header, the `| **Gate** |` row, AND the Approval History row the operator appends per the template's own convention. Every match emits its own 20-line window.
  2. The Approval History table sits directly above the UAT Sign-off + Attorney sections that BL-105/BL-115 ADDED to the personal template — sections that legitimately still carry `[Name — ...]` and `[YYYY-MM-DD]` placeholders at Phase 0 (they are Phase-3/4 evidence slots). The history-row window bleeds into them; the literal `YYYY-MM-DD` matches; WARN + issues+1 → gate BLOCKS.
  - Secondary trigger, same check: ANY bracketed annotation in the Reviewer cell of a genuinely filled entry (e.g. `[SIMULATED human role]`, or a real-world `Jane Doe [CTO]`) matches `Reviewer.*\[.*\]`.
- **Repro (clean):** fresh scaffold → fill the Phase 0→1 gate section completely (real name, 2026-07-18, "Approved") → add the matching row to `## Approval History` → `scripts/process-checklist.sh --start-phase1`.
- **Observed (verbatim, both runs):** `[WARN] Phase 0→1: APPROVAL_LOG.md entry contains placeholder values — fill in approver name and date` → `1 inconsistency(ies) found — blocking.` → `[FAIL] Phase 0→1 gate is NOT clear — start-phase1 refused`. Name and date WERE filled; the named remediation is wrong; nothing currently fillable fixes it (the trigger is future-phase template scaffolding).
- **Operator impact:** the honest path out is abandoning the Approval History convention (or hand-stripping future-phase placeholders — destroying template scaffolding). Walker resolved by removing the optional history row; gate-section entry (the canonical evidence) retained.
- **Suggested shape of fix:** anchor section extraction to the `^## Phase Gate:` header only (first match), stop the window at the next `^## `, and scope the placeholder regex to the entry's own table rows.
- **Positives in the same runs (not findings):** BL-114 HELD — `--start-phase1` consulted the gate and REFUSED to advance (F-DF2-003 fixed); gate-date auto-record worked and is idempotent (`preserving first-pass timestamp`); refusal did not clear recorded state; blocking-WARN increment semantics as documented.
- **Not a duplicate of:** BL-114 (rc-1 crash → diagnostics), BL-115 (date-row grammar/section), BL-105 (missing sections themselves).

## F-DF3-002 (HIGH) — The framework-generated CI runs check-phase-gate.sh, whose "Tools needed" arm BLOCKS in CI because dev-workstation tools (Semgrep/Snyk/Claude Code) are absent from the runner — the generated CI's governance gate is structurally unpassable
- **Where:** `scripts/check-phase-gate.sh` "Tool Resolution Check" block (grep-able: `Tools needed for Phase`), terminating `issues=$((issues + 1))` (~line 2072). Invoked by the generated `.github/workflows/ci.yml` step "Governance - Phase gate check" → `bash scripts/check-phase-gate.sh`.
- **Mechanism:** The block resolves required tools for the current phase via `scripts/resolve-tools.sh` reading `.claude/tool-preferences.json` context. `resolve-tools.sh` lists only tools MISSING from PATH. On a dev workstation Semgrep/Snyk/Claude Code are installed → empty list → no increment → gate passes. On the GitHub Actions ubuntu runner they are NOT installed (CI uses the `semgrep/semgrep-action` container for SAST and never uses Snyk or Claude Code) → the resolver returns all three `required:true` → `missing_count=3>0` → prints "Tools needed for Phase 1: Semgrep, Snyk CLI, Claude Code" → `issues+1` → "1 inconsistency(ies) found — blocking." → exit 1.
- **The tell:** the same block HAS CI-awareness — `prompt_yes_no` hard-N's the auto-install when `$CI` is set (line 142, deliberate: "eval install_command does NOT fire in CI"). So the code KNOWS it is in CI, declines to install, and then blocks anyway because the tools it declined to install are absent. The `issues+1` has no CI exemption while the install path does.
- **Repro (deterministic, real):** every push to `kraulerson/project-dogfood-3` whose earlier steps pass reaches this and fails. F3 commit `7af5a2d` CI run 29657490293: Build/Lint/Test/Semgrep/gitleaks/audit/license/lockfile/approval-log all ✓; **"Governance - Phase gate check" ✗** with exactly the output above. Locally `bash scripts/check-phase-gate.sh` rc=0 ("Phase gates consistent") because the tools are on PATH — the divergence IS the bug.
- **Why HIGH:** on the corporate tier, which can afford branch protection with required status checks, this gate is a required check that can NEVER pass in CI regardless of correct phase state — it hard-blocks every merge. On the personal free tier it produces a permanently-red governance job that trains the operator to ignore CI. There is no honest remediation inside the project: Snyk needs `SNYK_TOKEN` auth and Claude Code is an interactive agent CLI with no place in CI; installing them to satisfy a presence check is inappropriate. The only "fix" available to the operator is the FORBIDDEN `SOIF_PHASE_GATES=warn` — which I did NOT use.
- **Suggested fix:** the tool-availability check is a workstation-provisioning concern, not a phase-gate CONSISTENCY property. In CI (`[ -n "$CI" ]`/non-TTY) it should be skipped or made a non-blocking WARN (drop the `issues+1`), exactly mirroring the CI-awareness the auto-install prompt already has 100 lines above.
- **Cross-check:** NOT in the known-open set (BL-131/132 SAST, BL-135 bl033 flake, BL-136 full-suite debt, PR #215 zdr/sanitizer). No backlog entry covers the CI-invoked phase gate's tools check. NEW.
- **Walker's honest handling:** recorded, NOT worked around. Local gates (authoritative for phase progression) pass; CI governance job left red as the framework's defect. No forbidden escape used.

## F-DF3-004 (Medium) — Phase-2 user-terminal gate blocks legitimate NON-feat source commits: framework-gate.sh calls --check-commit-ready without --subject, defeating the documented subject short-circuit
- **Where:** `.git/hooks/framework-gate.sh` calls `"$SCRIPTS/process-checklist.sh" --check-commit-ready` with NO `--subject`. `check_commit_ready` (scripts/process-checklist.sh, `# code-process-checklist-5`) only bypasses the Phase-2 `require_build_loop_state_for_commit` block when `--subject` is present AND non-feat; absent a subject it defaults `subject_is_feat=true` and calls the build-loop-required block for ANY staged source file (`\.ts$|\.py$|src/|...`).
- **Effect:** At `current_phase=2` on a strict project, a user-terminal `git commit` with a legitimate non-feat type (`test:`, `chore:`, `refactor:`, `build:`, `ci:`, `perf:`, `style:`, `revert:`) that touches any source file is HARD BLOCKED ("[FAIL] pre-commit gate: 'feat(...)' commit blocked — no Build Loop active") unless a Build Loop happens to be active. The block fires at PRE-COMMIT (framework-gate), so the commit-msg surface — the only place `.git/COMMIT_EDITMSG` is current (BL-119) and where the subject COULD be read — is never reached.
- **The tell (code-process-checklist-5 is dead on this path):** the short-circuit was explicitly added "to unblock chore/fix/refactor/docs/test/perf/style/build/ci/revert source commits during Phase 2, matching the BL-006 classifier semantics." It works when given the subject but framework-gate never supplies it.
- **Proof (identical staged .ts files — playwright.config.ts, tests/e2e/reader.spec.ts):**
  - `process-checklist.sh --check-commit-ready` (framework-gate path, no subject) → **rc=1 (blocks)**
  - `process-checklist.sh --check-commit-ready --subject "test(e2e): x"` (commit-msg path) → **rc=0 (allows)**
  - `process-checklist.sh --check-commit-ready --subject "feat(x): y"` → rc=1 (correctly blocks feat)
  Real `git commit -m "test(e2e): ..."` at phase 2 emitted the feat-block after the pre-commit `[OK] semgrep` line and aborted (HEAD unchanged).
- **Why it matters:** the primary user-terminal enforcement path can't make a legitimate Phase-2 `refactor:`/`fix:`/`test:` source commit. `test:`/E2E work has a natural Phase-3 home, but a mid-Phase-2 `refactor:` or `fix:` to a `.ts` file does not — the operator is stuck choosing between a dishonest feat-reclassification+fake-build-loop, `--no-verify` (forbidden), or making the change docs-only.
- **Suggested fix:** move the Build-Loop-state gate to the commit-msg surface (where the subject is current), OR have framework-gate derive the prospective subject and pass `--subject` so the existing code-process-checklist-5 short-circuit engages on the terminal path too.
- **Cross-check:** not in BL-118..133, BL-135/136, PR #215. Related to but distinct from BL-119 (which removed the STALE-message consumer from --terminal-mode); this is the *inverse* gap — a subject-DEPENDENT decision that now runs with NO subject on the terminal path. NEW.
- **Walker's honest handling:** NOT worked around. E2E integration testing is Phase-3 Step 3.1 work; advanced to Phase 3 via the real 2→3 gate (Phase 2 genuinely complete) and committed the E2E suite there, where the Phase-2 build-loop block does not apply. No reclassification, no fake build loop, no --no-verify.

## F-DF3-005 (Medium) — BL-070 zap-dast leg can't retrieve its report under Colima/macOS: container writes it, Colima's bind mount never syncs it back, so the driver FAILs (not SKIPs) a verifiably-clean web app — and BL-130 then (correctly) forbids attesting that FAIL
- **Where:** `scripts/run-phase3-validation.sh` zap-dast scanner (`# BL-070-ZAP-DISPATCH`): `docker run --rm -v "$zap_tmp:/zap/wrk" ghcr.io/zaproxy/zaproxy:stable zap-baseline.py -t "$URL" -J zap-report.json`, then reads `$zap_tmp/zap-report.json`.
- **Mechanism (reproduced exhaustively):** the ZAP container's default `zap` user (uid 1000) cannot write the bind-mounted `/zap/wrk` (macOS `mktemp -d` is mode 700; `chmod 777` does not help — Colima's virtiofs mount does not honor host chmod for container writes) → `AccessDeniedException /zap/wrk/zap-report.json`. Running `--user root` DOES write the 24 KB report INSIDE the container (`-rw-r--r-- 1 root root 24478 /zap/wrk/zap-report.json`), but Colima never syncs that container→host write to the mounted macOS dir (host dir stays empty). Net: the driver's `$zap_tmp/zap-report.json` is always absent → `[FAIL] zap-dast — OWASP ZAP produced no report (rc=2) — treat as a scan failure, not a skip`.
- **The trap:** because the driver classifies "no report" as a FAIL (not a SKIP), BL-130 then refuses to attest it: `[FAIL] --attest REFUSED: 'zap-dast' last recorded a REAL FAIL ... a FAIL must be FIXED or RE-RUN, not attested`. So on macOS+Colima — a runtime the framework's own init/CLAUDE.md assume — the DAST leg has NO path to green through the driver even for a verifiably-clean app.
- **Proof the app is actually clean:** I ran the identical ZAP baseline manually and extracted the report via `docker cp` (which reads the container FS directly, bypassing the broken mount). BL-122 filter `jq '[.site[].alerts[] | select((.riskcode|tonumber)>=2)] | length'` = **0** after hardening (report: `docs/test-results/2026-07-18_zap_pass.json`; only 2 riskcode-0 informational alerts remain).
- **Impact:** any web-app project run on macOS+Colima hits a hard, un-attestable zap-dast FAIL at the Phase 3→4 gate regardless of the app's actual DAST posture. Workaround requires running ZAP out-of-band and attesting the *skip* (only possible if `SOLO_ZAP_TARGET_URL` is left unset so the leg SKIPs instead of FAILs — a non-obvious dance).
- **Suggested fix:** the driver should either (a) run the ZAP container as a user that can write the bind mount and/or `docker cp` the report out of a non-`--rm` container instead of relying on bind-mount write-back, or (b) distinguish "ZAP ran but the report file could not be retrieved (environment)" from "ZAP found blocking findings" so the former is an attestable SKIP, not a BL-130-locked FAIL.
- **Cross-check:** distinct from BL-122 (that fix — riskcode≥2 filter — is correct and HELD, verified above) and BL-130 (also correct; it's the interaction that traps). Not in BL-131/132/135/136 or PR #215. NEW.
