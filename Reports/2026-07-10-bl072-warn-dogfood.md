## What "would-block" means here

A commit is counted as a would-block if its subject is `feat:`/`fix:`/
`refactor:` AND its own changed files contain at least one implementation
file (anything not under `tests/ docs/ .github/ Reports/ templates/` and
not `scripts/lint-*.sh`) AND no test file rides along in the same commit.

### ⚠️ These counts are an UPPER BOUND on the live false-block rate

Per-commit replay classifies each commit on **its own files only**
(`git diff-tree`). It cannot reproduce the live gate's allowance that a
test written **earlier on the same branch** (`git diff main...HEAD`)
satisfies the commit. On this repo, features were routinely split across
commits on a PR branch (tests in one commit, implementation in the next),
so many replay would-blocks would pass live. Treat every number below as a
ceiling, not the live rate.

It is also path-based only: a `feat/fix/refactor` commit that changes just
comments in an implementation file still counts as impl (a documented C1
limitation). WARN-only mode makes both over-counts harmless.

## Recommendation to Karl (⚖️ decision — do NOT build C2 yet)

**The measured false-block rate is too high to ship a naive same-commit hard
block.** My defended reading of the numbers:

- **Raw would-block rate:** 38.6% of feat/fix/refactor commits (upper bound;
  the live rate is lower because the branch-allowance carve-out is not applied
  in per-commit replay).
- **Clean false-positive floor:** ~50% of would-blocks (hand-verified on the
  20 most-recent) are the gate being simply *wrong* — non-code misclassified as
  impl, deletions, refactors, or already-tested code.
- **Practical unhelpful-block rate:** closer to **~70–80%** of would-blocks
  once you add the integration/`init.sh`/host-driver "true positives" where the
  correct response is an **attestation**, not a test-first block — the code was
  validated, just not by a fast-lane unit test.

So a hard block that fires on "impl file with no test in the same commit" would
stop or nag roughly 2 in 5 legitimate commits on this repo, and about three
quarters of those stops would be noise. That is exactly the brittleness the
backlog's own push-back flagged.

**If C2 proceeds, these are non-negotiable (all deferred to the C2 work, not
built here):**

1. **Keep the branch-allowance** — the single biggest gap between replay
   (upper bound) and live. It must survive into any hard-block form.
2. **Tighten the classifier before it can block** — exclude, at minimum:
   repo-root `*.md` (backlog, README, HANDOFF, CHANGELOG), **deletion-only**
   diffs, and copy-list/scaffolding-shaped changes. ~50% of the false-block
   rate is this misclassification alone.
3. **Tier-key the block on `deployment` + `poc_mode` (BL-084 `# BL-084-TIER-KEY`
   pattern), NOT `track`** — Personal / POC-Personal WARN-and-log; only
   POC-Sponsored / Production hard-block.
4. **Ship the `SOLO_TDD_ATTESTED=1` escape** (attested, recorded to
   `process-state.json::tdd_attestations[]`) at the same time as the block —
   the `init.sh` / host-driver / integration surface is structurally not
   fast-lane-unit-testable and will need it on nearly every such commit.
5. Consider routing `init.sh`, `scripts/host-drivers/*`, and
   `scripts/upgrade-project.sh` to an "integration-attested" lane rather than
   demanding a same-commit unit test that the framework's own architecture
   makes impractical.

**Phase C1 as shipped (WARN + ledger + this replay) is the safe, correct stop.**
The live WARN + `.claude/tdd-warn-ledger.jsonl` now accrue real-world data on
in-flight commits; revisit the hard-block decision once that ledger has a few
weeks of signal to corroborate this historical replay.
