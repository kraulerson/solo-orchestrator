> This report is the headline evidence of what the C2 hardening bought. It measures the
> **classifier** change in isolation (BEFORE = the original C1 impl rule; AFTER = the tightened
> C2 rule), over one fixed commit window, so the delta is not confounded by `main` advancing

## What C2 changed (three levers)

1. **Classifier tightening** (`scripts/lib/tdd-classify.sh`) — the only lever a per-commit
   replay can measure. Excludes, before the gate can ever block:
   - **all `*.md` anywhere** (backlog, README, HANDOFF, CHANGELOG, notes) — the single biggest
     C1 misclassification class;
   - **pure file deletions** (`git name-status` status `D`) — removing a source file ships no
     implementation;
   - **lockfiles / generated manifests** (`package-lock.json`, `*.lock`, `yarn.lock`).
2. **Tier-keying** on `deployment` + `poc_mode` (`# BL-084-TIER-KEY`, never `track`) — only
   **sponsored-POC / production** tiers hard-block; **personal / private-POC** WARN-and-log.
   Invisible to this replay (the replay has no tier).
3. **Attestation escape** `SOLO_TDD_ATTESTED=1` (recorded to
   `.claude/process-state.json::tdd_attestations[]`) — turns the structurally-not-unit-testable
   integration surfaces (`init.sh`, `scripts/host-drivers/*`, `scripts/upgrade-project.sh`)
   from a block into a recorded exception. Also invisible to this replay.

## Why the *live* hard-block rate is far below 36.4%

The 36.4% is a **classifier upper bound on a Full-shaped repo**, and it is a ceiling three times
over:

1. **Per-commit replay has no branch-allowance.** The live gate treats a test written earlier on
   the same PR branch (`git diff main...HEAD`) as satisfying the commit; the replay classifies
   each commit on its own files. This repo routinely split tests and implementation across
   sibling commits, so a large fraction of the 106 would pass live. (Unchanged from C1.)
2. **Tier-keying exempts personal / private-POC entirely.** On those tiers the gate WARNs and
   logs `bypassed:true` — it never blocks. Only sponsored-POC / production hard-block.
3. **The attestation escape covers the integration surfaces.** As the hand-review below shows,
   ~9 of the 11 remaining true-positives are `init.sh` / host-driver / `upgrade-project.sh`
   integration code that is validated by UAT / scenario / integration rather than a fast-lane
   unit test. On a sponsored/production tier those commit with `SOLO_TDD_ATTESTED=1` + a reason,
   which is **recorded, not silenced** — a block that becomes an auditable exception, not a wall.

So the residual "block a legitimate commit with no recourse" rate is near zero: it is only the
handful of cases (below: `250d6e1`, `5167164`) where a quick same-commit unit test was genuinely
the right missing artifact — exactly the cases the gate *should* stop.

## Conclusion

The classifier tightening does exactly what it was scoped to do — it removes the pure
**misclassification** false-positive class (markdown, file deletions, lockfiles), a 1.7-point
drop on this window. It cannot (and does not try to) remove the refactor / branch-allowance /
integration false-positives; those are handled by the other two hardening levers that a
per-commit replay cannot show: the **branch-allowance** (live), **tier-keying** (personal/POC
WARN-only), and the **attestation escape** (integration surfaces → recorded exception). Together
these make the sponsored/production hard block land on the ~2-in-20 commits that genuinely shipped
untested runtime code, while every other case is either WARNed-and-logged, satisfied by a
sibling-commit test, or attested-and-recorded — never a silent pass, never a wall.
